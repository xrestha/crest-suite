## D. Known roadmap items

- [x] ~~S440 offline-sync hardening pass~~ — 2026-07-22, found during a whole-app audit/debug pass (see `README.md` S440). Two fixes on the POS offline order-taking path, a sibling to S372's own `flushPosOrderQueue` idempotency fix: (1) `enqueuePosOrder` in `offlineQueue.js` did a read-modify-write across *two* IndexedDB transactions (readonly `get` → merge → readwrite `put`). `saveOrder` fires `logKotSend('KOT')` + `logKotSend('BOT')` un-awaited back-to-back, both routing through it while offline — so two concurrent calls read the same `kot_sends:[]` pre-image and the second `put` clobbered the first, silently dropping one station's queued KOT/BOT send so it never replayed into `pos_kot_log` on sync (KOT Reconciliation under-counts; the physical ticket still printed, so no *order* was lost). Fixed by collapsing get+merge+put into a single `readwrite` transaction — IndexedDB serialises overlapping readwrite transactions on a store, so the append is now atomic regardless of await ordering. (2) Offline orders minted their permanent `pos_orders.id` from `crypto.randomUUID()`, `undefined` on WebViews older than Chrome 92 / Safari 15.4 — would throw mid-save, lose the order, and hang the save button. Added `src/utils/uuid.js` (native, falling back to a `getRandomValues` v4) and wired it in. Re-confirmed billing stays hard-gated offline (`payDisabled` includes `!isOnline`) — the offline surface is order-taking only, no money path. Build clean, 91/91 tests pass.
- [x] ~~Discount netting bug + Stock Movements ledger report~~ — shipped S395, 2026-07-15, found via a cross-module (IMS↔POS↔HR) audit. Two related fixes: (1) `writeSalesEntries()` in `PosOrders.jsx` snapshotted `unit_price` on `'pos'`-source `sales_entries` rows at full pre-discount price regardless of the bill's `discountAmt`/`discRatio` — meaning every IMS revenue report reading that column (Monthly/Annual/Period Comparison Summary, Menu Repricing/Engineering, Recipe Margin, Overheads, Best Sellers, Sales.js, Owner Dashboard's Food Cost %/Net Margin %) silently overstated revenue by the discount total on every discounted bill, since S286's `pos_comp` source-exclusion fix only handled comps, never discounts. Fixed by applying the same `discRatio` already computed for the bill's own VAT recalc to `'pos'`-row `unit_price` (only on `closeType === 'paid'`); `'pos_comp'` rows stay full-price since they're already revenue-excluded and only `qty_sold` is read downstream. (2) `stock_movements` (the POS depletion ledger from S228/IMS stock deduction trigger) turned out to have exactly one reader anywhere in the app — Reorder Report's Book Stock column — with no way to see the ledger itself. New `/stock-movements` report (Starter, `stock_movement_log` flag) shows the full per-period ledger: item, qty depleted, source (POS Sale/Comp), the causing order (click-through via the existing `viewPosBill()` helper), staff, and food-cost value; a "Comp Value" stat card surfaces what complimentary items actually cost — a number that existed nowhere else despite comps correctly being zero-revenue everywhere else. Reorder Report's Book Stock cell now links through, pre-filtered by item/period.

- [x] ~~Delivery-partner payment tagging (Foodmandu/Pathao)~~ — shipped S290, 2026-07-07; reworked S291, 2026-07-07 after real-world testing showed the S290 design was wrong. S290 modeled Foodmandu/Pathao as two new top-level `payment_method` values, as if they paid the restaurant at the counter. They don't — both platforms collect from the customer and remit to the restaurant later (weekly/monthly), minus commission, which is a receivable exactly like Credit, not an instant payment method. S291 reworks this: `payment_method` reverts to just Cash/Card/eSewa/Khalti/FonePay/Credit/Split (dropped the two aggregator values from the CHECK constraint), and a new `pos_orders.delivery_partner` column (`Foodmandu`/`Pathao`/`NULL`) tags a Credit bill's buyer instead. At Charge, selecting Credit reveals a Foodmandu/Pathao quick-select chip that auto-fills `buyer_name`/`buyer_phone` (fixed sentinel phone numbers per platform, so `pos_customers` groups them consistently) — no commission math happens at Charge at all, per explicit client direction that staff shouldn't see/estimate commission while billing. Commission is now entered only at settlement time, in Customers → Outstanding Credit → Settle: the commission % (still configured in Table Management → Delivery Partners, `settings.pos_foodmandu_commission_pct`/`pos_pathao_commission_pct`) pre-fills as a suggestion against the bill's ex-VAT (taxable) base, adjustable against the platform's actual remittance before confirming, then stored on `pos_orders.commission_amount`. Sales Report gets a new 10th tab, Delivery Partners (replacing the Commission/Net Received columns S290 had added to Payment Summary/Bill Register, both reverted since commission is no longer a payment-method-level concept) — one row per Foodmandu/Pathao bill, Outstanding until settled, then showing commission and net actually received. Two real bugs found and fixed while wiring up S290 still stand: (1) `PosShifts.jsx`'s X/Z report had its own hardcoded copy of the payment-method list (not imported from the shared constant) that would have silently dropped aggregator-tagged bills from shift totals — moot now since these bills are just Credit again, but the shared-constant fix stays; (2) `settings` table RLS only allowed admin to INSERT/UPDATE, so every settings-writing tab in Table Management (Discounts, Quick Notes, Ticket Routing, and this one) had been silently no-op'ing for any real client login — fixed at the policy level. No live order-injection/menu-sync with either platform — that's the separate, still-deferred item above. Same-day follow-up, still S291: the platform list itself (names, commission %, sentinel phone) was also hardcoded to exactly Foodmandu/Pathao — made fully client-editable instead (`settings.pos_delivery_partners` jsonb array of `{name, commission_pct, phone}`, replacing the two fixed `pos_foodmandu_commission_pct`/`pos_pathao_commission_pct` columns), since aggregator partnerships come and go and a new one shouldn't need a code change. Table Management → Delivery Partners is now an add/rename/remove list (same UX pattern as Discount Reasons) instead of two fixed input pairs; `pos_orders.delivery_partner`'s CHECK constraint relaxed from `ANY('Foodmandu','Pathao')` to any non-empty string. Plus three small live-use polish fixes: added **Cheque** and **Bank Transfer** settle methods (how a delivery platform/corporate credit actually remits — `credit_settled_method` CHECK relaxed; settlement-only, not in `PAYMENT_METHODS`); fixed the settlement row rendering the platform name twice ("Foodmandu Foodmandu") and in a too-small badge font; and fixed the Collected/Sales-Report Commission+Net columns gating on `commission_amount > 0` (a platform bill settled at 0% showed "—/—" like missing data — now gates on "is a delivery-partner bill", so it reads NPR 0 commission / full net received).
